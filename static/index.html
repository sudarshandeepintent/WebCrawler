Batch Crawl
Submit up to 50 URLs at once — they're fetched concurrently so the whole batch finishes in roughly the time of one slow request. Use this when you already know which pages you want. URLs already in cache return immediately without hitting the network.
Deep Crawl
True BFS web crawler — give it a starting URL and it automatically follows links, discovering new pages level by level. You control how deep it goes (max depth) and how many pages it visits (max pages). Enable "Stay on domain" to keep it from wandering off to other sites.

How each crawl mode works, when to use which one, and how the system handles caching and topic scoring under the hood.

The Three Crawl Modes

Single Crawl
You give it one URL. It fetches that page, extracts everything — metadata, links, images, topics — and returns it. That's it. Nothing is followed automatically.
Batch Crawl
You give it a list of URLs you already know. All of them are fetched at the same time (up to the concurrency limit). Good when you have a specific set of pages to analyse.
Deep Crawl
You give it one starting URL. It fetches it, finds all the links on that page, fetches those, finds more links, and keeps going — automatically — until it hits your depth or page limit.

How Deep Crawl Works (BFS)

Deep Crawl uses Breadth-First Search. It processes all pages at depth 1 before moving to depth 2, then all depth 2 before depth 3, and so on. This means shallower, more important pages are always crawled first.

Depth 0: seed URL Depth 1: all links found on seed Depth 2: all new links found on depth-1 pages ...and so on until max_depth or max_pages is reached

A visited set tracks every URL that's been seen. If the same URL appears on multiple pages at different depths, it's only crawled once — the first time it's encountered. This prevents infinite loops and duplicate work.

URL normalization strips fragments (#section) and trailing slashes so /page and /page/ are treated as the same URL.

Stay on domain — when enabled, any link pointing to a different domain is ignored. rei.com and www.rei.com are treated as the same domain so you don't miss internal links just because of a www prefix.

How Caching Works

Every successful crawl result is stored in the cache keyed by URL. On the next request for the same URL, the cached result is returned immediately — no network call, no parsing, no classification. This is why second requests are near-instant.

The memory cache is an LRU (Least Recently Used) cache — when it fills up (default 500 entries), the entry that was accessed least recently gets evicted first. Each entry also has a TTL (default 1 hour) after which it expires automatically.

Set REDIS_URL in your environment to switch to Redis. Useful for production deployments where multiple instances run in parallel.

How Topic Scoring Works

Each page is scored against 16 topics: technology, outdoors, finance, e-commerce, news, education, entertainment, travel, sports, politics, science, food, real-estate, automotive, healthcare, fashion.

The scoring process:

  • Tokenize — the page text (title, description, headings, body) is broken into unigrams, bigrams, and trigrams so phrases like "machine learning" match as a unit.
  • TF scoring — each keyword match is weighted by its term frequency (how often it appears relative to total tokens). Long pages don't get an unfair advantage over short ones.
  • Exclusivity weight — a keyword that only belongs to one topic gets 3× weight. A keyword shared by two topics gets 1×. Shared by three or more gets 0.4×. This prevents generic words like "research" from inflating multiple topics equally.
  • Threshold — topics with a score below 0.08 are dropped from results to avoid noise.

Page category is assigned separately — first by URL path hints (/blog/, /shop/, /docs/), then by body signals ("add to cart", "posted by", "breaking news"), then falls back to the top-scoring topic.

Why Some Sites Return 403 (and How It's Handled)

Sites like Amazon, REI, and others protected by Akamai or Cloudflare inspect the TLS fingerprint of every connection. Python's default HTTP libraries have a fingerprint that's instantly recognisable as non-browser traffic, so they return 403 before even looking at your request.

This crawler uses curl-cffi with impersonate="chrome124" to mimic Chrome 124's TLS handshake at the binary level. Combined with a full Chrome header set (including Sec-Fetch-* and Sec-CH-UA headers), the requests look identical to a real browser from the server's perspective.

If curl-cffi isn't available, it falls back to httpx with retry logic for 429/503 responses.
